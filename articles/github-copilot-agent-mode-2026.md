## はじめに

GitHub Copilot、気づいたらコード補完ツールではなくなっていた。

Agent Mode が GA になり、CLI には `/fleet` で複数エージェントを同時に走らせる機能が載り、SDK でアプリに組み込めるようになった。ここ数ヶ月で触ってきた中で、実際に開発の進め方が変わったと感じたポイントを書いていく。

## Agent Mode — IDE の中で動くもう一人の開発者

従来の Copilot は、書きかけのコードの続きを補完するか、チャットで質問に答えるか。どちらかだった。Agent Mode はここから一歩踏み込んで、「自分で計画を立てて、ファイルを跨いで編集して、テストまで回す」という動きをしてくれる。

やっていることはシンプルで、ワークスペースを見渡して関連ファイルを拾い、コードを書き換え、ターミナルでコマンドを叩き、エラーが出たら直す——このループを自律的に繰り返す。

### 実践例：レガシーコードのモダナイズ

たとえば古い Python スクリプトを整理したいとき、こう投げる。

```text
このディレクトリ内の古いデータ処理スクリプトを、最新のPandasと型ヒントを使ったモダンなPythonコードにリファクタリングして。pytestのユニットテストも追加して。
```

対象ファイルの特定から、リファクタリング、テスト作成、実行確認まで一気にやってくれる。途中でテストが落ちれば自分で修正に入る。

## Copilot CLI と `/fleet` — ターミナルから並列で回す

Copilot CLI もだいぶ変わった。中でも `/fleet` コマンドが面白い。タスクを投げると、オーケストレーターが作業を分解して複数のサブエージェントに並列で振り分けてくれる。

```bash
copilot /fleet "APIモジュールのドキュメントを作成してください。
- docs/authentication.md (認証フロー)
- docs/endpoints.md (エンドポイント仕様)
- docs/errors.md (エラーコード)
- docs/index.md (上記3つへのリンク、他のタスク完了後に実行)"
```

依存関係を明示しておけば、独立したタスクは同時に走り、依存するものは順番に処理される。ドキュメント生成みたいな「量はあるけど独立した作業」にはかなり効く。

### カスタム命令で出力の質を揃える

チームで使うなら `.github/copilot-instructions.md` は早めに整備しておきたい。コーディング規約やプロジェクト構成を書いておくだけで、Copilot の出力がブレにくくなる。

`.github/prompts/docstring.prompt.md` のようにタスク特化のプロンプトファイルを用意すれば、定型作業をコマンド一発にまとめられる。

## Copilot SDK — アプリにエージェントを埋め込む

テクニカルプレビューの段階ですが、Copilot SDK を使うと CLI の裏側で動いているのと同じ実行ループ（計画→ツール呼び出し→マルチターン実行）に、自分のアプリからアクセスできます。

社内向けの AI アシスタント GUI を作ったり、特定の業務フローに特化した自動化ツールを組んだり。Copilot の頭脳を借りつつ、インターフェースは自由に設計できるのが面白いところです。

MCP サーバーとの統合もサポートされていて、GitHub の MCP サーバーをつなげばリポジトリや Issue、PR の操作までエージェントに任せられます。

## まとめ

Agent Mode、CLI の `/fleet`、SDK。やれることが増えた分、どこから手をつけるか迷うかもしれません。

個人的には、まず Agent Mode でコード修正やテスト追加を数回試してみるのがいいと思います。Copilot の癖がわかってきたら CLI で環境構築やドキュメント生成を並列で流してみる。SDK はまだプレビューですが、社内ツールを作りたい人には触りがいがあるはずです。

カスタム命令（`.github/copilot-instructions.md`）を早めに整備しておくと、どの機能を使っても出力の質が安定するので、ここは先にやっておいて損はありません。
