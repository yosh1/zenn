## AIエージェントは「使い方」より「環境」で決まる

Claude Code、Codex、Devin。選択肢は増えた。ただ、どれだけ賢いモデルを持ってきても、エージェントが動く「環境」が雑だと成果はブレる。

LangChainのチームが面白い実験をしている。モデルを一切変えず、ハーネス（エージェントの外側にある環境設計）だけを手直しして、SWE-benchスコアを52.8%から66.5%まで引き上げた。Anthropicも同じモデルでハーネス次第で約10ポイント精度が変わると報告している。

ここでは、2026年4月時点の情報をもとに、ハーネスエンジニアリングを**3つのレイヤー**に分けて実装する方法を書いていく。

```
┌─────────────────────────────────────────────┐
│  CLAUDE.md / AGENTS.md                      │
│  → エージェントに「何をすべきか」を教える    │
├─────────────────────────────────────────────┤
│  CI/CD + マルチエージェント                  │
│  → 「どう動かすか」を定義する               │
├─────────────────────────────────────────────┤
│  LiteLLM + LangFuse                         │
│  → 「うまくいっているか」を計測する          │
└─────────────────────────────────────────────┘
```

Martin Fowlerが2026年4月2日に公開した記事の用語を借りると、上からガイド（フィードフォワード制御）、センサー（フィードバック制御）、計測レイヤー。この3つが揃って初めて、エージェントが安定して動く環境になる。

## ハーネスエンジニアリングとは何か

HashiCorp創業者のMitchell Hashimotoが2026年2月に定義した概念で、OpenAI Codexチームの技術記事をきっかけに広まった。

> エージェントがミスを犯したとき、そのミスを二度と繰り返さないよう解決策をエンジニアリングする行為

プロンプトエンジニアリングからコンテキストエンジニアリング、そしてハーネスエンジニアリングへ。粒度がどんどん広がってきた。プロンプトエンジニアリングは「個々の指示文の最適化」、コンテキストエンジニアリングは「コンテキストウィンドウ全体の設計」。ハーネスエンジニアリングはそこからさらに一歩引いて、**エージェントを取り巻くシステム全体**を設計する。

OpenAIの公式記事によると、Codexチームの7人は5ヶ月で約100万行のコードを1,500のPR経由で生成した。人間が手書きしたソースコードはゼロ。それを支えたのは、依存関係の一方向制約や構造テストによるレイヤリング違反の自動検出。ドキュメントディレクトリの機械的な維持管理もそう。全部モデルの「外側」にある仕組みだ。

## 前提：プロダクトごとに「どこまでAIに任せるか」を決める

3つのレイヤーに入る前に、もっと手前の設計判断がある。**そもそもどこまでAIに任せるか**をプロダクト単位で決めること。

全部のプロダクトを同じ温度感でAI化しようとすると、まず間違いなくコケる。コードベースの歴史が違えば、障害時のビジネスインパクトも違う。チームのAI習熟度だってバラバラだ。ハーネスの設計が変わるのは当然だろう。

### AI委譲の3段階

| レベル | 意味 | 人間の関与 | 適するプロダクト |
|--------|------|-----------|----------------|
| **フルAI開発** | 実装からレビュー、マージまでAIが主導 | 方向性の判断のみ | グリーンフィールド、壊れても致命的でない領域 |
| **領域分離型** | 機能のクリティカル度でAI/人間を切り替え | クリティカルな変更は人間レビュー必須 | 事業の根幹だが、全機能が等しく重要ではない場合 |
| **段階的移行** | 限定的な範囲でAI導入し、信頼を積み上げてから拡大 | 初期は人間が主導 | レガシーコードベース、AIの挙動を予測しづらい領域 |

たとえば、同じプロダクトでも決済処理と管理画面のダッシュボードでは、障害時のインパクトが桁違いだ。決済はサーバー停止もデータ欠損も許されないから人間レビュー必須。ダッシュボードは多少止まっても致命傷にはならないからフルAI化。この切り分けをCI/CDのワークフローに落とし込む。ディレクトリやファイルパスのパターンでジョブを分岐させて、クリティカルなパスを含むPRには人間のApproveを必須にすればいい。

### レガシーコードベースの扱い

歴史の長いコードベースを、いきなりフルAI開発に切り替えるのは無理がある。コンテキストウィンドウを食い尽くすし、暗黙の依存関係をAIが拾えない。クロスリポジトリの変更が頻発する環境だと、構造的にハードルが高い。

やるなら「小さく始めて信頼を積む」。PRトリアージの自動化、定型的なバグ修正のAI化、ドキュメント生成あたりから手をつけて、計測データを見ながら範囲を広げていく。

### エンジニアのコンセンサス

技術的な設計より厄介なのが、チーム内の合意だ。AIに任せたくないエンジニアは普通にいる。コードへのオーナーシップもあるし、AI出力への不信感もある。自分の仕事が減る不安だってある。理由はいろいろだけど、トップダウンで「全部AIでやれ」と押し通すと現場が荒れる。

うまくいくのは、チーム単位で委譲レベルを合意して、各エンジニアが自分の担当領域でどこまでAIに委ねるかを選べる余地を残すやり方だ。そこに計測データ（後述するLangFuseのポジティブ評価）が積み上がってくると、「ここまでは任せて大丈夫だったよね」と実績ベースでレベルを引き上げられる。

強制ではなくデータで納得してもらう。この順番が逆になると、ハーネスをどれだけ丁寧に設計しても現場に根付かない。

## レイヤー1：CLAUDE.md / AGENTS.md

エージェントに「何をすべきか」を伝えるファイル群。ハーネスの中ではガイド（フィードフォワード制御）にあたる。

![ハーネスのGuides/Sensorsと3つの規制カテゴリ（Maintainability、Architecture Fitness、Behaviour）の関係](/images/harness-types.png)
*出典：Martin Fowler, "[Harness engineering for coding agent users](https://martinfowler.com/articles/harness-engineering.html)"*

### 書くべきもの・書くべきでないもの

Hashimotoは自身のGhosttyプロジェクトのAGENTS.mdについて「各行は過去のエージェントの悪い振る舞いに基づいている」と述べている。裏を返せば、エージェントが間違えないことは書く必要がない。

ETH Zurichの2026年3月の研究がこれを裏付けている。LLMが推論できる情報（一般的なコーディング規約など）を書き連ねると、ノイズになってかえって性能が落ちる。「これを消したらエージェントがミスするか？」と自問して、そうでなければ削る。判断基準はそれだけでいい。

Anthropic公式の推奨は**200行以内**（上限目安300行）。コードスタイルはリンターに任せて、LLMにリンターの仕事をさせない。

### CLAUDE.md の設計指針

Claude Codeが参照するプロジェクトの説明書。Progressive Disclosureパターンが肝で、全情報を事前に与えるのではなく、重要情報の「見つけ方」を教えてJust-in-Timeでアクセスさせる。

```markdown
# プロダクト名

## 概要
[1-2文。AIがプロダクトの目的を正しく把握するために必要な最低限]

## 技術スタック
[言語、フレームワーク、インフラ。推論不可能な情報に絞る]

## アーキテクチャ
[ディレクトリ構成、レイヤー構造]
※独自パターンを採用している場合のみ詳述

## コマンド
- テスト: `npm run test`
- ビルド: `npm run build`
- リント: `npm run lint`

## このプロダクト特有のルール
[AIが知らなければ間違える情報のみ]
- 例: 外部APIの仕様でLLMが誤解しやすいポイント
- 例: 複数リポジトリ間の依存関係と変更の順序
- 例: 触ってはいけないファイル・ディレクトリ

## 参照ドキュメント
@docs/architecture.md
@docs/api-spec.md
```

`.claude/rules/` 配下にMarkdownを置くと自動的にCLAUDE.mdと一緒に読み込まれるので、ルールの分割管理もできる。チーム共通のルールと個人の好みを分けるのに使える。

### AGENTS.md の設計指針

OpenAI Codexが自動検出して階層的に命令を組み立てるファイル。CLAUDE.mdとは別ファイルとして管理する。

Codexの探索順序は `AGENTS.override.md` → `AGENTS.md` → `TEAM_GUIDE.md` → `.agents.md` で、グローバル設定（`~/.codex/AGENTS.md`）とリポジトリローカル、さらにサブディレクトリ単位で細分化できる。上限は32KiB。

```markdown
# AGENTS.md

## レビュー基準
- severity: CRITICAL / IMPORTANT のみ対応。NITSは自動修正か無視
- スコープ外の指摘は除外（差分に含まれないコードへの言及禁止）

## 修正ルール
- NITSレベル: 自動修正 → コミット → プッシュ
- IMPORTANT以上: 人間に判断を委ねる

## ディレクトリ別ルール
- `src/payments/`: 入力バリデーション必須、PCI-DSS準拠チェック
- `src/api/`: OpenAPI仕様との整合性確認
```

### 両方持つべきか

Claude CodeとCodexを併用するなら、CLAUDE.mdとAGENTS.mdの両方を持つことになる。内容の重複は避けて、共通のルールはどちらか一方に寄せて、もう一方からポインタを張る形が現実的だろう。

## レイヤー2：マルチエージェントCI/CD

エージェントを「どう動かすか」のレイヤー。エージェントが行動した後に結果を検知して、自己修正を促す仕組みだ。

### 2026年4月のツール状況

主要なツールが出揃って、マルチエージェント構成が現実的な選択肢になった。

**Claude Code**

Hooksで21のライフサイクルイベントに介入できる。`PreToolUse`でアクションのブロック、`PostToolUse`で自動検証。サブエージェントとGit Worktreeの統合で、複数エージェントの並行開発がコンフリクトなしでできるようになった。Agent Teams（実験的機能）を使えば、チームリードがタスクの調整から結果統合まで行う。GitHub Actions公式の `anthropics/claude-code-action@v1` もGA済み。

**OpenAI Codex**

CLIから画像添付ができる（スクリーンショット、ワイヤーフレーム）。GitHub Actions公式の `openai/codex-action@v1` があり、CI失敗時の自動修正パターンも公式に掲載されている。AGENTS.mdの自動検出と階層的な命令組み立てが特徴。

**Devin**

PlannerとCoder、Criticの3モデル構成（Compound AIシステム）。Issueを割り当てると実装からテスト、PR作成、レビュー対応まで自律で回す。PR merge rateが34%から67%に改善したと報告されている。

**GitHub Agentic Workflows（テクニカルプレビュー）**

Agent HQでClaude、Codex、Copilotを同一タスクに同時割り当てできる。

![GitHub Agent HQ：Copilot、Claude、Codexをドロップダウンから選択できる](/images/github-agent-hq-picker.jpg)

### なぜ複数エージェントか

単一エージェントに全工程を任せるのは、もう得策ではない。

Qodoの研究では、複数の特化エージェント（論理エラー、ベストプラクティス、エッジケース、クロスファイル依存）を並列投入するマルチエージェント方式が、F1スコア60.1%を達成してシングルエージェントを9%上回った。

OpenAI Codexチームも、エージェント間コードレビューを採用している。Codexが自身の変更をレビューし、別エージェントに追加レビューを依頼する構成だ。人間が介入するのは新規アーキテクチャ判断やセキュリティ、プロダクト方針に限定している。

### GitHub Actionsでの構成例

PRオープン時にClaude CodeとCodexを並列で走らせる最小構成。

```yaml
name: Multi-Agent Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  claude-review:
    runs-on: ubuntu-latest
    steps:
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            Review this PR focusing on architecture and domain-specific logic.
            Refer to CLAUDE.md for project context.
          claude_args: "--max-turns 5"

  codex-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: openai/codex-action@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          prompt: |
            Review this PR for security vulnerabilities, performance issues,
            and edge cases. Follow AGENTS.md guidelines.
          sandbox: "read-only"
```

Devinを加える場合は、Issue起票をトリガーにする。Biz側がIssueを作成、Devinが実装PR作成、上記のレビューワークフローが走る、という流れになる。

### 得意領域で使い分ける

| ツール | 得意な仕事 | 渡し方 |
|--------|-----------|--------|
| **Devin** | スコープが明確なタスク実行（Issueから実装、PR作成まで） | Issue割り当て、Slackリアクション |
| **Codex** | 網羅的なバッチレビュー、ルールベースの検査 | PR作成時に自動実行 |
| **Claude Code** | コンテキストが複雑な修正、設計判断、対話的な操作 | @claudeメンション、ローカルCLI |

全部のPRにフルセットを回す必要はない。PRの複雑度に応じてエージェントの数を調整する。小さなconfig修正にマルチエージェントレビューは過剰で、コストだけが膨らむ。

### オシレーション（振り子問題）への対策

マルチエージェント構成で避けて通れない問題がある。エージェントAの修正をBが元に戻し、それをAが再修正する無限ループ。トークン消費が指数的に増えて、成果物はゼロなのに外部からは「作業中」に見える。

対策のベストプラクティスが固まりつつある。

**どう検出するか**

エージェント出力の状態ハッシュを取り、95%類似の応答が3回繰り返されたら脱出シーケンスを発動する。ここで大事なのは「エージェントにループしているか聞くことはできない。数学的に証明しなければならない」という原則。自己診断ではなく、外部からの機械的ガードレールが必要になる。

**どう防ぐか**

- `MAX_ITERATIONS=8` の上限を設ける。リトライ前に「何が失敗したか？何を変えるか？」の強制リフレクションを挟む
- 3回スタックしたらタスクを別エージェントに再割り当て
- セッションごとのトークン/コスト予算にハードシーリングを設定する（例：$5.00）
- 軽量なSLM（Small Language Model）を監視専用に配置して、繰り返しの不一致や停滞を検知する「サーキットブレーカーエージェント」を置く

## レイヤー3：LiteLLM + LangFuse による計測

「うまくいっているか」を定量的に把握するレイヤー。ここが抜けると、施策の効果を体感でしか語れなくなって、チームへの横展開も経営への報告も難しくなる。

### アーキテクチャ

LiteLLM ProxyをLLM APIのゲートウェイとして立てて、全呼び出しをLangFuseにトレースとして送信する。2026年の推奨構成はOpenTelemetry（OTEL）ベース。

```yaml
# litellm_config.yaml
model_list:
  - model_name: claude-sonnet-4-6
    litellm_params:
      model: claude-sonnet-4-6
  - model_name: gpt-5.4
    litellm_params:
      model: gpt-5.4

litellm_settings:
  callbacks: ["langfuse_otel"]
```

```bash
export LANGFUSE_PUBLIC_KEY="pk-lf-..."
export LANGFUSE_SECRET_KEY="sk-lf-..."
export LANGFUSE_OTEL_HOST="https://us.cloud.langfuse.com"

litellm --config litellm_config.yaml
```

これだけで、全LLM呼び出しのトレース（入出力、レイテンシ、トークン数、コスト）がLangFuseのダッシュボードに流れ始める。

:::message
LiteLLM + LangFuseの具体的なセットアップ手順や構成の選び方は、別記事「[LiteLLM × Langfuse を安く安定して動かす構成（VPS + Cloudflare）](https://zenn.dev/preferred/articles/litellm-langfuse-hybrid-setup)」で詳しく書いた。本記事ではハーネスエンジニアリングにおける計測の考え方に集中する。
:::

### ポジティブ評価のみの設計

ここが計測レイヤーの肝になる。

ネガティブ評価（「ここがダメだった」を記録する方式）は直感的に正しく見えるけど、実運用だと問題を起こしやすい。減点方式が続くとチームは萎縮する。ガードレールが過剰になって、エージェントの自律性を制限する方向に倒れがちだ。

代わりに、**良かった出力にだけスコアをつけて蓄積する**アプローチを取る。スコアが付かないトレースは「未評価」として扱い、減点はしない。

LangFuseのスコアリングは数値型、ブール型、カテゴリ型に対応していて、この設計は標準的なパターンで実装できる。

```typescript
// PRレビューで人間がApproveした指摘を記録
langfuse.score({
  traceId: reviewTraceId,
  name: "human-approved",
  value: 1,
  comment: "Reviewer accepted this finding",
});
```

```python
# 自動マージ後にN日間障害が起きなかったPRを記録
langfuse.create_score(
    trace_id=merge_trace_id,
    name="stable-merge",
    value=1,
    comment=f"No incidents for {days} days after merge"
)
```

### 何を計測するか

| 計測対象 | トリガー | 見えてくるもの |
|---------|---------|-------------|
| 人間がApproveしたAIレビュー指摘 | PR Approve時に自動記録 | 「良い指摘」のパターン |
| 自動マージ後に障害が起きなかったPR | マージ後N日経過で自動記録 | 安全にマージできる条件 |
| RSIループで成功したリファクタ | テスト通過+マージで記録 | 効果的な改善パターン |
| エージェント間のApprove（A→Bレビュー通過） | ワークフロー内で自動記録 | エージェント構成の相性 |

蓄積されたポジティブ評価は「何がうまくいくか」のナレッジベースになる。定期的にダッシュボードを振り返って、スコアが高いパターンを強化し、スコアが付かないパターン（人間にもAIにも認められていない）を見直す。

### LLM-as-a-Judgeとの併用

人手での評価には限界があるから、自動評価も組み合わせる。LangFuseにはフルマネージドのLLM-as-a-Judge評価機能がある。

2026年のベストプラクティスは「1つの失敗モードに焦点を絞り、Pass/Failで判定する」設計。「コードの品質を総合的に評価して」みたいな曖昧な指示ではなく、「この修正はスコープ外のコードを変更していないか？」のように具体的な観点に絞る。

ただ、導入の初期フェーズでは人間のフィードバック収集に集中するのが現実的だ。自動評価の精度は人間の評価データを土台にして高めていくもので、順番が逆になると信頼性の低い自動評価に振り回される。

## 3つのレイヤーを回す

3つのレイヤーは単独でも使えるけど、組み合わせると改善のループが回り始める。

1. **CLAUDE.md / AGENTS.md** でエージェントに方向性を与える
2. **マルチエージェントCI/CD** でエージェントの出力を検証する
3. **LiteLLM + LangFuse** で何がうまくいったかを記録する
4. 計測結果をもとに **CLAUDE.md / AGENTS.md を更新** する

4が1に戻る。Hashimotoの言葉を借りれば「エージェントがミスを犯したとき、二度と繰り返さないよう環境を直す」。その「直す」をデータで裏付けてやるのが、計測レイヤーの役割だ。

Stripeはこのサイクルを回して週1,300以上のPRを自動マージしている。OpenAIのCodexチームは5ヶ月で100万行を人間の手書きなしで生成した。規模は違うけど、仕組みの骨格は同じだ。

## 導入の順序

全部を一度に入れようとすると頓挫する。段階を踏んだほうがいい。

### フェーズ0：AI委譲レベルの合意

技術的な設計に入る前に、プロダクトごとの「どこまでAIに任せるか」を決める。前述の3段階（フルAI開発 / 領域分離型 / 段階的移行）のどれにあたるかをチーム内で合意しておく。ここが曖昧なまま施策を走らせると、「ここはAIに任せていいのか？」という判断が個人に委ねられて、チームの中で基準がバラつく。

### フェーズ1：CLAUDE.md / AGENTS.md の整備（1-2週間）

コストゼロで始められて、効果がすぐに体感できる。まず1つのリポジトリで200行以内のCLAUDE.mdを書く。すでにAIツールを使っているメンバーの暗黙知を掘り起こす作業にもなる。

### フェーズ2：CI/CDへのAIレビュー導入（2-3週間）

PRオープン時にClaude CodeかCodexのレビューを走らせる。最初はシングルエージェントで十分。ワークフローのYAMLは先述の例が最小構成になる。フェーズ0で決めた委譲レベルに応じて、自動マージまで許可するか、人間Approveを必須にするかを分ける。

### フェーズ3：LiteLLM + LangFuse の計測基盤（1-2週間）

フェーズ2のレビューが動き始めたら、トレースを仕込む。計測対象が存在しない状態で計測基盤だけ作っても意味がないので、この順番が大事になる。

### フェーズ4：マルチエージェント化と評価ループ（継続）

計測データが溜まってきたら、エージェントの追加やルールの調整をデータに基づいて行う。ここから先は「回す」フェーズで、終わりのないプロセスだ。段階的移行のプロダクトは、ポジティブ評価の蓄積に応じてAI委譲レベルを引き上げていく。

## まとめ

やることはシンプルで、**AI委譲レベルを決めて**から**3つのレイヤーを組み立てる**。

プロダクト特性に応じて「どこまでAIに任せるか」を先に決める。そこが曖昧だと、どんなにハーネスを丁寧に作っても現場が迷う。

レイヤーは、CLAUDE.md / AGENTS.mdでエージェントが間違えることだけを書き、マルチエージェントCI/CDで得意領域ごとにツールを使い分け、LiteLLM + LangFuseでポジティブ評価だけを記録する。

モデルは来月にはもっと賢くなっている。でも、ハーネスはモデルが変わっても使い回せる。ここに時間を投資する価値は、今後ますます大きくなっていくと思う。
